APIScout 系统架构文档

客户系统接入方案生成器 — 给我一个 URL，还你一份 OpenAPI spec

版本 0.1.0 | 2026-03-30 | dingjq + Claude 协作

1. 产品定位

核心价值：不管客户系统什么技术栈、有没有 API 文档，都能快速生成接入方案。
核心体验：你正常操作系统，APIScout 在后台默默录制，操作完给你一份 API 文档。

APIScout 是 insight68 客户实施的核心工具。insight68 需要对接客户的各种企业系统（EAM/MES/ERP），但每个客户的系统技术栈不同、有没有 API 文档也不确定。APIScout 解决的是"快速摸清客户系统有哪些 API"这个问题。

2. 七条发现路径

APIScout 不绑定任何一种技术方案。给什么条件，走什么路径：

#路径输入输出状态
1客户直接给 Swagger 文件JSON/YAML 文件OpenAPI + Swagger UI已实现
2系统有在线 Swagger 端点URL（自动探测）OpenAPI + Swagger UI已实现
3框架 metadata（Jmix 等）适配器自动探测OpenAPI + Swagger UI已实现
4HTTP 抓包（浏览器录制）用户操作系统OpenAPI + Swagger UI已实现
5数据库 schema 扫描DB 连接字符串Schema 报告（交互式 HTML）+ 交叉增强 spec已实现
6源码分析项目源码AI 直接读源码生成 spec规划中
7系统文档 + schema + AI文档 + DBAI 增强 SQL 模板V2

4. 核心模块详解

capture/ 网络录制层

recorder.py — Playwright 响应监听，挂载到浏览器页面

filter.py — 黑名单模式：录一切，只排除已知第三方（Google/百度/CDN）和框架噪声（Vaadin UIDL）

store.py — JSONL 增量写盘，每条记录立即持久化，崩溃安全

analyzer/ 分析推断引擎

router.py — 路径参数化归并。支持 UUID/雪花ID/MongoDB ObjectId/编码ID/数字ID/Hash 六种模式，保留词排除（api/v1/日期）

schema_engine.py — genson 多次合并 + format 检测（date-time/email/uuid）+ enum 检测

auth_detector.py — Bearer JWT / Basic / API Key / Cookie / 金蝶 / 用友非标

dedup.py — 端点聚合 + query 参数推断

crawler/ 页面探索

navigator.py — BFS 页面队列 + SPA 菜单自动点击（通用 DOM 扫描，不依赖特定 UI 框架）+ 安全按钮探索 + 智能停止

api_prober.py — 探测 15 个已知 API 文档端点

link_extractor.py — DOM 链接提取（含 hash 路由）

js_analyzer.py — 6 种正则从 JS 源码提取 API URL

adapters/ 框架适配器

registry.py — @register 装饰器注册，detect_framework() 自动匹配

jmix.py — /rest/metadata/entities → 完整 CRUD spec（实战验证：48 端点）

spring_boot.py — /actuator + /v3/api-docs → 下载 spec

ruoyi.py — /dev-api/swagger-resources → 下载 spec

加新框架 = 加一个文件 + @register，不改旧代码

db_scanner/ 数据库扫描 V1.1 新增

connector.py — 连接管理 + 方言自动检测（连接字符串 → 方言实例）

introspector.py — 扫描编排（表 → 列 → 索引 → 外键 → 枚举探测 → 关系推断）

sampler.py — 枚举探测（双门槛 + 置信度分级 high/medium/low）+ 随机采样

relations.py — 关系推断（xxx_id → xxx.id + 同模块匹配 + 用户表特殊处理）

dialect/ — 4 种方言：PostgreSQL / MySQL / Oracle / SQL Server

交互式审核报告 V1.2 新增

schema_report_html.py — 单文件自包含 HTML（数据 + JS 交互全部内嵌）

推断关系/枚举候选支持 ✓/✗ 审核 + 目标表修改 + 枚举值增删

按表分组折叠 + 搜索框 + 状态筛选 + 进度条

导出/导入审核结果 JSON — 发给客户审核，客户发回 JSON

schema_enricher.py — 用 DB schema 交叉增强 OpenAPI spec

5. 数据流

用户操作浏览器
↓
recorder 监听所有响应
↓
store → capture.jsonl（增量写盘）
↓
analyzer（路由 + schema + 认证 + 聚合）
↓
generator（OpenAPI + Swagger UI + 报告）

6. 与 insight68 的接口

客户情况APIScout 输出insight68 Tool 类型
有 REST APIOpenAPI specrest_api（调 API）
有 Swagger 文件导入 → OpenAPI specrest_api
有框架 metadata适配器 → OpenAPI specrest_api
有数据库Schema 报告 + 交叉增强 specrest_api（更精确）
只有数据库（V2）Schema + 文档 → SQL 模板集db_query（查 DB）
两种 Tool 对 insight68 用户完全透明 — "帮我查设备列表" 背后走 API 还是 SQL，用户不关心。

7. 实战验证

momExecution

技术栈：Jmix + Vaadin

发现方式：适配器（metadata）

结果：48 端点，24 实体

教训：Vaadin 抓包抓不到 REST

gtshebei.com

技术栈：Vue SPA

发现方式：手动录制

结果：51 端点，bearer_jwt

教训：SPA 菜单无 href

eamNge

技术栈：若依 + Vue（跨端口 CORS）

发现方式：手动录制

结果：37 端点

教训：前后端不同端口

eamNewGe DB 扫描 V1.1

技术栈：若依 + PostgreSQL

发现方式：DB schema 扫描（apiscout db）

结果：185 表 / 3240 列 / 92 外键 / 795 推断关系

枚举检测：置信度分级（参考 ydata-profiling 双门槛）

输出：交互式审核 HTML（分组折叠 + 搜索 + 导入导出）
