# APIScout V1.2 — 交互式审核报告

> 日期：2026-03-30 | 作者：dingjq + Claude 协作

## 1. 目标

把静态 HTML 报告变成**可交互的审核工具** — 用户在浏览器里对推断关系和枚举候选做 ✓/✗ 确认，审核完导出 JSON。

**核心使用场景：**
```
实施人员：apiscout db → 生成 schema_report.html（单文件，自包含）
         → 邮件/微信发给客户

客户：    浏览器打开 → 审核 ✓/✗ → 点"导出"
         → 得到 eam_db_review_20260330.json（几 KB）
         → 发回给实施人员

实施人员：拿到 review JSON → 后续 AI 增强 / insight68 对接
```

客户零安装、零学习成本，打开就能用。

## 2. 设计决策

| 决策 | 结论 | 理由 |
|------|------|------|
| 核心目标 | 审核交互 | 直接解决"用户如何给反馈"的痛点 |
| 入口 | HTML 报告本身 | 最灵活，可发给任何人 |
| 存储 | JSON 文件 | 不引入 SQLite，V1.3 再考虑 |
| 数据嵌入 | 全部数据嵌在 HTML 里 | 单文件发送，不依赖 schema_report.json |
| 审核粒度 | ✓/✗ + 少量可编辑 | 覆盖 90% 场景 |
| 后端改动 | 只改 HTML 模板生成 | 纯前端 JS，不动 API |

### 3.2 审核操作

**推断关系：**
- ✓ 确认 → 变绿，关系被采纳
- ✗ 拒绝 → 变红划线，关系被排除
- 目标表可点击修改 → 弹出下拉框选择其他表（候选列表来自全部表名）
- 修改后自动标为 ✓ 确认

**枚举候选：**
- ✓ 确认 → 整组变绿
- ✗ 拒绝 → 整组变红划线
- 单个值可 ✗ 删除（点击值旁的小叉）
- 可 + 添加新值（弹出输入框）
- 删除/添加后自动标为 ✓ 确认（已编辑）

**状态颜色：**
- 未审核：默认样式（灰色边框）
- 确认：绿色背景 + ✓ 标记
- 拒绝：红色划线 + ✗ 标记
- 已编辑：绿色背景 + ✎ 标记

### 3.3 工具栏操作

- **进度条**：实时显示 已审核数 / 总数
- **导出审核结果**：下载 `{database}_review_{date}.json`
- **导入审核结果**：上传之前导出的 JSON，恢复审核状态
- **全部确认**：一键确认所有未审核项（快速通过，后面再挑错的拒绝）
- **全部拒绝**：一键拒绝所有未审核项
- **重置**：清除所有审核状态

## 4. 数据嵌入方案

HTML 文件中嵌入完整数据为 JS 变量：

```html
<script>
// 完整扫描数据嵌入（替代外部 schema_report.json）
const SCHEMA_DATA = {
  dialect: "postgresql",
  database: "eam_db",
  scanned_at: "2026-03-30T12:00:00Z",
  tables: [...],              // 完整的 TableInfo 列表
  explicit_relations: [...],  // 显式外键
  inferred_relations: [...],  // 推断关系
};

// 审核状态（内存中维护）
const REVIEW_STATE = {
  relations: {},  // key: "source_table.source_column" → {status, target_table, target_column}
  enums: {},      // key: "table.column" → {status, values: [...]}
};
</script>
```

### 4.1 HTML 生成

`schema_report_html.py` 的改动：
- `generate_schema_report_html(report)` 改为同时嵌入 `SCHEMA_DATA` 和 JS 交互代码
- 人工核查区的 HTML 从纯展示改为带交互控件
- Jinja2 模板负责 HTML 结构，JS 负责交互逻辑
- 所有 JS 代码内联（不依赖外部 .js 文件）

### 4.2 文件大小预估

- 当前 schema_report.html：~5MB（185 表的采样数据占大头）
- 加入 JS 交互代码后：~5.5MB（JS 约几百行，几十 KB）
- 单文件，可以邮件/微信直接发

## 6. 导入恢复逻辑

用户打开新报告 → 导入之前的 review JSON → 自动匹配恢复：
- 按 `source_table.source_column` 匹配推断关系
- 按 `table.column` 匹配枚举候选
- 新增的推断（上次没有的）标为"未审核"
- 已删除的推断（上次有现在没有的）忽略

## 7. 技术实现

### 7.1 改动范围

只改一个文件：`apiscout/core/generator/schema_report_html.py`

- Jinja2 模板加入交互控件（按钮、下拉框、输入框）

  - 导入：`<input type="file">` + `FileReader` + JSON.parse

## 8. 不做的事

- 不引入 SQLite
- 不改后端 API
- 不改 CLI 命令
- 不做多人协作（一份 HTML 一个人审核）
- 不做实时保存（审核完手动导出）
- 不做表结构编辑（表/字段信息不可改，只有推断结果可审核）
- 不改 schema_report.json 的生成逻辑（JSON 文件保持不变，交互只在 HTML 里）

## 9. 后续消费

审核结果（schema_review.json）的消费方：
- `schema_enricher.py`：读取 confirmed 的关系和枚举，增强 OpenAPI spec
- 未来 AI SQL 模板生成：只用 confirmed 的表关系和枚举定义
- insight68 对接：只导出 confirmed 的数据

消费时的规则：
- status=confirmed → 采纳
- status=rejected → 忽略
- 未出现在 review JSON 中的 → 当作 unreviewed，按场景决定是否采纳
