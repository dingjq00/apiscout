# APIScout V1.1 — 数据库 Schema 扫描设计

> 日期：2026-03-30 | 作者：dingjq + Claude 协作

## 1. 目标

为 APIScout 新增第 5 条发现路径：**数据库 Schema 扫描**。

两个核心场景：
- **A. 独立报告** — 客户系统没有 API（或抓不到），只给 DB 只读账号。扫 schema → 出报告。
- **B. 交叉增强** — V1 已录制到 API，用 DB schema 的精确类型、注释、关系增强 OpenAPI spec。

## 2. 设计决策

| 决策点 | 结论 | 理由 |
|--------|------|------|
| 核心场景 | A + B 都做 | 覆盖面最广 |
| 数据库类型 | PG / MySQL / Oracle / SQL Server | 客户环境混杂 |
| 连接方式 | 直连（VPN/SSH 用户自己处理） | 零额外开发 |
| 输出格式 | V1.1 先出 schema 报告（JSON + HTML），SQL 模板格式后面再定 | 先看到结果再定模板 |
| 扫描范围 | 全扫所有表 | 判断太复杂，客户自己也不清楚哪些需要 |
| 技术方案 | 原生 SQL 直查 information_schema（方案 B） | 信息最全（含注释），依赖最轻，SQL 透明 |

## 3. 架构

### 3.1 新增模块

```
apiscout/core/
├── db_scanner/               ← V1.1 新增

│   ├── introspector.py       # schema 提取 + 关系推断
│   ├── sampler.py            # 样本数据采集（枚举探测 + 随机采样）
│   └── dialect/              # 方言 SQL

│       ├── postgresql.py
│       ├── mysql.py
│       ├── oracle.py         # ALL_TAB_COLUMNS 而非 information_schema
│       └── mssql.py
├── analyzer/
│   └── schema_enricher.py    ← V1.1 新增：用 DB schema 增强 OpenAPI spec
├── generator/

└── workflow.py               ← 扩展：加 db_scan 工作流
```

### 3.2 与现有架构的关系

```
V1 路径:  crawler → capture → analyzer → generator → OpenAPI spec
                                  ↑
V1.1 路径: db_scanner ──────────→ schema_enricher（交叉增强）
                  ↓
              generator → schema_report（独立报告）
```

- `db_scanner/` 跟 `crawler/` + `capture/` 平级，不互相依赖
- 交叉增强通过 `schema_enricher.py` 桥接，不侵入 V1 代码
- 独立报告通过新增的 `schema_report_html.py` 生成

## 4. 数据模型

### 4.1 核心结构

```python
@dataclass
class ColumnInfo:
    name: str
    data_type: str              # 原始数据库类型（varchar(255), int4, NUMBER(10,2)）
    normalized_type: str        # 统一类型：integer / string / number / boolean / date-time
    nullable: bool
    default: str | None
    comment: str | None         # 字段注释 — AI 生成模板的关键输入
    max_length: int | None
    numeric_precision: int | None
    is_primary_key: bool
    enum_values: list[dict] | None  # [{"value": "ACTIVE", "count": 500}, ...]

    @property
    def is_enum_candidate(self) -> bool:
        return self.enum_values is not None


@dataclass
class IndexInfo:
    name: str
    columns: list[str]
    is_unique: bool
    is_primary: bool


@dataclass
class ExplicitRelation:
    """数据库中真实存在的外键约束"""
    source_table: str
    source_column: str
    target_table: str
    target_column: str
    constraint_name: str


@dataclass
class InferredRelation:
    """通过命名模式推断的关系"""
    source_table: str
    source_column: str
    target_table: str
    target_column: str
    confidence: float
    evidence: str               # "命名匹配: device_id → device.id"


@dataclass
class TableInfo:
    name: str
    schema: str                 # 统一语义：命名空间（PG=public, MySQL=dbname, Oracle=owner）
    comment: str | None
    row_count: int | None       # 估算值（不跑 COUNT(*)）
    columns: list[ColumnInfo]
    indexes: list[IndexInfo]
    sample_rows: list[dict]     # key 统一小写，ORDER BY RANDOM() LIMIT 20


@dataclass
class SchemaReport:
    dialect: str                # postgresql / mysql / oracle / mssql
    database: str
    scanned_at: str             # UTC ISO 时间戳
    tables: list[TableInfo]
    explicit_relations: list[ExplicitRelation]
    inferred_relations: list[InferredRelation]

    @property
    def total_tables(self) -> int:
        return len(self.tables)

    @property
    def total_columns(self) -> int:
        return sum(len(t.columns) for t in self.tables)

    @property
    def total_foreign_keys(self) -> int:
        return len(self.explicit_relations)

    @property
    def total_inferred_relations(self) -> int:
        return len(self.inferred_relations)

    @property
    def enum_candidates(self) -> int:
        return sum(1 for t in self.tables for c in t.columns if c.is_enum_candidate)
```

### 4.2 设计要点

- **`enum_values` 带 count**：不只知道有什么值，还知道分布，AI 能判断主态
- **统计字段用 property**：从 tables 算出来，不存储，避免不一致
- **列名统一小写**：方言层负责，消费方不关心数据库大小写差异
- **关系提到全局**：SchemaReport 顶层统一管理，不散在各 TableInfo 里

## 5. 扫描策略

### 5.1 元数据提取

每种方言实现同一套接口：

```python
class BaseDialect:
    def get_tables(conn) -> list         # 表清单 + 注释 + 估算行数
    def get_columns(conn, table) -> list  # 字段 + 类型 + 注释 + 主键
    def get_constraints(conn) -> list     # 外键约束（全局）
    def get_indexes(conn, table) -> list  # 索引
    def sample_rows(conn, table, n) -> list          # 随机采样
    def distinct_values(conn, table, col, limit) -> list  # 枚举值
    def count_distinct(conn, table, col) -> int       # 基数检测
    def normalize_type(data_type: str) -> str         # 类型归一化
```

核心方言差异：

| 差异 | PG / MySQL / SQL Server | Oracle |
|------|------------------------|--------|
| 元数据来源 | `information_schema.*` | `ALL_TAB_COLUMNS`, `ALL_CONSTRAINTS`, `ALL_COL_COMMENTS` |
| 随机采样 | `ORDER BY RANDOM() LIMIT 20` | `SAMPLE(1)` 或 `WHERE ROWNUM <= 20` |
| 行数估算 | PG: `pg_stat_user_tables`; MySQL: `TABLE_ROWS`; MSSQL: `sys.partitions` | `ALL_TABLES.NUM_ROWS` |

### 5.2 枚举探测（两步走）

**第一步：基数检测**
```sql
SELECT COUNT(DISTINCT col) FROM table
```
- 基数 < 50 → 枚举候选
- 基数 >= 50 → 普通字段

**第二步：拿值 + 分布**
```sql
-- 枚举候选字段
SELECT col, COUNT(*) as cnt FROM table GROUP BY col ORDER BY cnt DESC

-- 整体样本（非枚举字段的上下文）
SELECT * FROM table ORDER BY RANDOM() LIMIT 20
```

### 5.3 关系推断

显式外键从 `information_schema.key_column_usage` / `ALL_CONSTRAINTS` 直接拿。

隐式关系推断规则（按置信度排序）：

| 模式 | 例子 | 置信度 |
|------|------|--------|
| `xxx_id` → `xxx.id` 且 xxx 表存在 | `device_id → device.id` | 0.9 |
| `parent_id` → 同表 `.id` | 树结构自引用 | 0.85 |
| `created_by` / `updated_by` → 用户表 | 审计字段 | 0.8 |
| `xxx_code` → `xxx.code` 且字段存在 | 编码关联 | 0.7 |

推断规则设计为可扩展列表，不硬编码。

## 6. 交叉增强

`analyzer/schema_enricher.py` 的职责：

```
输入：OpenAPI spec (V1 生成) + SchemaReport (V1.1 扫描)
处理：
  1. 用 DB 精确类型替换 genson 推断的模糊类型
  2. 用 DB comment 补充 OpenAPI description
  3. 用枚举值补充 enum 字段
  4. 用外键/推断关系补充 x-relations 扩展字段
输出：增强后的 OpenAPI spec
```

匹配策略：API 端点的路径段 + 响应字段名 → DB 表名 + 列名。模糊匹配，不要求精确。

## 7. 命令设计

```bash
# 基本用法 — 连接字符串
apiscout db "postgresql://readonly:pass@192.168.1.100:5432/eam"
# → output/eam/schema_report.json + schema_report.html

# 分开指定参数
apiscout db --host 192.168.1.100 --port 5432 --user readonly \
            --password pass --database eam --dialect postgresql

# 交叉增强已有 spec
apiscout db "postgresql://..." --enrich output/eam/openapi.yaml
```

输出目录自动从数据库名生成，不暴露 `-o` 参数。

## 8. 输出

### 8.1 文件结构

```
output/<database>/
├── schema_report.json     # 完整扫描结果（SchemaReport 序列化）
├── schema_report.html     # 人可读的可视化报告
└── openapi.yaml           # （如果 --enrich）增强后的 spec
```

### 8.2 HTML 报告内容

- 数据库概览：方言、表数、字段数、关系数
- 每张表：字段列表（类型、注释、是否主键、是否枚举）
- 枚举值展示：值 + 分布
- 关系图：显式外键（实线）+ 推断关系（虚线）
- 样本数据预览

### 8.3 JSON 格式

SchemaReport dataclass 直接 `dataclasses.asdict()` 序列化，方便后续 AI 消费。

## 9. 依赖

不引入 SQLAlchemy。只加数据库驱动：

| 数据库 | Python 驱动 | 备注 |
|--------|------------|------|
| PostgreSQL | `psycopg2-binary` | 最常用 |
| MySQL | `mysql-connector-python` | 纯 Python，免编译 |
| Oracle | `oracledb`（原 cx_Oracle） | Oracle 官方维护 |
| SQL Server | `pymssql` 或 `pyodbc` | pymssql 更轻 |

安装策略：驱动按需安装，不全装。
```bash
pip install apiscout[postgresql]   # 只装 psycopg2
pip install apiscout[mysql]        # 只装 mysql-connector
pip install apiscout[db-all]       # 全装
```

## 10. 不做的事（V1.1 边界）

- 不做 SQL 模板生成 — 先出报告，模板格式跟 insight68 对接时再定
- 不做数据脱敏 — 后续按需加
- 不做 SSH 隧道 — 用户自己处理网络
- 不做表过滤 — 全扫
- 不做写操作 — 全部只读（SELECT + information_schema）
- 不做 Web 面板集成 — V1.1 先 CLI，后续加

## 11. 与 insight68 的接口（预留）

V1.1 先出报告，V2 加 SQL 模板时的输出矩阵：

| 客户条件 | APIScout 输出 | insight68 Tool 类型 |
|---------|-------------|-------------------|
| 有 REST API | OpenAPI spec | rest_api |
| 只有数据库 | schema 报告 → (V2) SQL 模板 | db_query |
| API + 数据库 | 交叉增强 OpenAPI spec | rest_api（更精确）|
